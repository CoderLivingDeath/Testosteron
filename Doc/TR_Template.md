# 2. Цели и назначение создания системы

## 2.1 Цель создания системы

**Целями** создания системы являются:

1. Предоставление платформы для проведения онлайн-тестирований любой сложности
2. Упрощение процесса создания и редактирования тестов администраторами
3. Обеспечение возможности прохождения тестов без регистрации для гостевых пользователей
4. Хранение результатов тестирования для зарегистрированных пользователей
5. Обеспечение гибкости структуры тестов с поддержкой различных типов вопросов

## 2.2 Назначение системы

**Система предназначена для:**

- Создания тестов с различными типами полей (текстовые, радио-кнопки, чекбоксы)
- Управления тестами через административную панель
- Прохождения тестов пользователями через веб-интерфейс
- Хранения и отображения результатов тестирований
- Аутентификации и авторизации пользователей с разделением ролей (admin, user)
- Формирования гибкой структуры тестов с помощью JSON-поля

# 3. Характеристика объекта автоматизации

## 3.1 Общая характеристика объекта автоматизации

Система онлайн-тестирования «Testosteron» представляет собой веб-приложение для создания, управления и прохождения тестов.

**Основные характеристики:**

- **Тип системы:** Веб-приложение (SPA/MVC гибрид)
- **Архитектура:** Трехуровневая (Presentation → Business Logic → Data Access)
- **Пользователи:** Неограниченное количество (масштабируемо)
- **Режим работы:** Круглосуточный (24/7)

## 3.2 Существующие бизнес-процессы

**Бизнес-процессы, подлежащие автоматизации:**

1. **Создание теста:**
   - Администратор создает новый тест
   - Добавляет название, описание
   - Добавляет поля вопросов с различными типами (text, radio, check)
   - Сохраняет тест

2. **Прохождение теста:**
   - Пользователь открывает тест по уникальной ссылке (GUID)
   - Заполняет поля ответов
   - Отправляет результат
   - Получает подтверждение о сохранении

3. **Просмотр результатов:**
   - Зарегистрированный пользователь может просмотреть свои результаты
   - Администратор может управлять всеми тестами

## 3.3 Существующая инфраструктура

- **Платформа:** .NET 10 (ASP.NET Core MVC)
- **СУБД:** PostgreSQL (рекомендуется), SQLite (для разработки)
- **ORM:** Entity Framework Core 10
- **Аутентификация:** ASP.NET Core Identity
- **Веб-сервер:** Kestrel (встроенный) / Nginx (прокси)
- **Фронтенд:** Razor Views, Bootstrap 5, jQuery

# 4. Требования к системе

## 4.1 Требования к системе в целом

### 4.1.1 Требования к структуре

Система должна иметь MVC-архитектуру с выделением следующих компонентов:

**Клиентская часть (Presentation Layer):**
- Razor Views (.cshtml) для рендеринга страниц
- Bootstrap 5 для стилизации
- jQuery для клиентских сценариев
- AJAX-запросы для асинхронных операций

**Серверная часть (Business Logic Layer):**
- Контроллеры (Controllers) для обработки HTTP-запросов
- Сервисы (Services) для бизнес-логики
- DTO/ViewModels для передачи данных

**Доступ к данным (Data Access Layer):**
- Entity Framework Core для ORM
- Репозитории (Repositories) для абстракции доступа к данным
- Миграции (Migrations) для управления схемой БД

**Функциональные подсистемы:**

1. **Подсистема управления тестами (Admin Area):**
   - Создание тестов
   - Редактирование тестов
   - Удаление тестов
   - Предпросмотр тестов

2. **Подсистема прохождения тестов (Public Area):**
   - Просмотр списка тестов
   - Прохождение теста по GUID
   - Отправка ответов

3. **Подсистема аутентификации и авторизации:**
   - Регистрация пользователей
   - Вход в систему
   - Разграничение прав доступа
   - Управление ролями (admin, user)

### 4.1.2 Требования к численности и квалификации персонала

**Требования к численности персонала:**

| Роль | Количество | Функции |
|------|-----------|---------|
| Системный администратор | 1 | Развертывание, настройка, мониторинг |
| Контент-менеджер | 1-5 | Создание тестов |
| Конечные пользователи | Неограничено | Прохождение тестов |

**Требования к квалификации:**

- Системный администратор: опыт администрирования Linux и PostgreSQL от 1 года
- Контент-менеджер: базовые навыки работы с ПК, обучение 4 часа
- Конечный пользователь: навыки работы с браузером

### 4.1.3 Требования к надежности

- Время непрерывной работы системы: 99% (не более 7.3 часа простоя в месяц)
- Время восстановления после сбоя: не более 1 часа
- Автоматическое резервное копирование: ежедневно в 03:00
- Хранение резервных копий: 30 дней
- Журналирование операций: 365 дней

### 4.1.4 Требования к эргономике

**Требования к пользовательскому интерфейсу:**

- Интерфейс на русском языке
- Поддержка браузеров:
  - Google Chrome (последние 2 версии)
  - Mozilla Firefox (последние 2 версии)
  - Microsoft Edge (последние 2 версии)
  - Safari (последние 2 версии)
- Время загрузки страницы: не более 3 секунд (3G)
- Адаптивный дизайн для разрешений от 320×480 до 1920×1080
- Поддержка мобильных устройств (Bootstrap 5)

### 4.1.5 Требования к защите информации

**Требования к информационной безопасности:**

1. **Аутентификация:**
   - Хранение паролей с использованием PBKDF2
   - Минимальная длина пароля: 3 символа
   - Блокировка учетной записи после 5 неудачных попыток входа
   - Время блокировки: 5 минут

2. **Авторизация:**
   - Разграничение доступа по ролям (admin, user, guest)
   - Политика «что не разрешено, то запрещено»
   - Защита административной области

3. **Защита данных:**
   - HTTPS (TLS 1.2+) в production
   - Защита от SQL-инъекций (параметризованные запросы)
   - Защита от XSS (валидация ввода)
   - Защита от CSRF (Anti-Forgery Tokens)

4. **Журналирование:**
   - Логирование действий пользователей
   - Логирование ошибок
   - Хранение логов: 365 дней

### 4.1.6 Требования к производительности

- Одновременных пользователей: до 100 (базовый режим)
- Время отклика: не более 2 секунд для 95% запросов
- Обработка транзакций: до 1000 операций в час
- Масштабируемость: горизонтальное масштабирование (при необходимости)

### 4.1.7 Требования к переносимости

- Кроссплатформенность: Windows, Linux, macOS
- Контейнеризация: Docker (опционально)
- Независимость от конкретной СУБД (через Entity Framework)

  - Название вопроса (до 100 символов)
  - Тип вопроса (text/radio/check)
  - Описание (до 400 символов)
  - Опции (массив строк)
  - Обязательность поля
- Сохранение теста в формате JSONB

**F006 Редактирование теста:**

- Изменение параметров теста

**F007 Удаление теста:**

**F008 Просмотр результатов:**

- Отображение списка пройденных тестов
- Дата прохождения

**F009 Управление ролями:**

- Создание ролей (admin)

**F010 Предпросмотр теста:**

- Рендеринг теста как при прохождении

## 6.1 Виды испытаний системы

| Вид испытаний | Содержание | Критерий приемки |
|---------------|------------|------------------|
| Модульное тестирование | Тестирование отдельных компонентов (сервисы, контроллеры) | Покрытие > 70% |
| Интеграционное тестирование | Взаимодействие компонентов | Все сценарии работают |
| Системное тестирование | Тестирование всей системы | 100% требований выполнено |
| Приемочное тестирование | Проверка Заказчиком | Подписание акта |

## 6.2 Требования к приемке работ по стадиям

| Стадия | Участники | Критерии приемки |
|--------|-----------|------------------|
| Проектирование | Заказчик, Разработчик | Согласование архитектуры |
| Разработка | Разработчик | Код прошел ревью, тесты зеленые |
| Тестирование | QA, Разработчик | Нет критических багов |
| Внедрение | Заказчик, Разработчик | Система развернута и работает |

## 7.1 Технические мероприятия
