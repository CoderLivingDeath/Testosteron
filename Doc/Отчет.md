# ВВЕДЕНИЕ {#введение}

Современное образование невозможно представить без использования информационных технологий. Одним из ключевых инструментов контроля знаний является компьютерное тестирование, которое позволяет объективно оценить уровень подготовки обучающихся, обеспечивает удобство проведения экзаменов и зачетов, а также значительно экономит время преподавателей.

Актуальность темы курсовой работы обусловлена необходимостью автоматизации процесса создания и проведения тестов в образовательных учреждениях. Ручное создание тестов требует значительных временных затрат от преподавателей, а использование готовых решений не всегда удовлетворяет специфическим требованиям конкретных дисциплин. Разработка собственного веб-приложения для проведения тестирования позволит создать гибкий инструмент, адаптированный под нужды конкретного образовательного учреждения.

**Целью** курсовой работы является разработка веб-приложения «Testosteron» для проведения тестирования, позволяющего создавать тесты с различными типами вопросов, проходить тестирование и просматривать результаты.

Для достижения поставленной цели были сформулированы следующие **задачи**:

1. Изучить теоретические основы разработки информационных систем и принципы тестирования программного обеспечения.

2. Провести анализ требований к разрабатываемой системе.

3. Спроектировать архитектуру и базу данных информационной системы.

4. Реализовать веб-приложение с использованием современных технологий.

5. Провести тестирование разработанного приложения.

6. Составить документацию и руководство пользователя.

**Объектом** исследования является процесс создания и проведения компьютерного тестирования.

**Предметом** исследования является веб-приложение «Testosteron» для проведения тестирования.

В первой главе работы рассматриваются теоретические основы разработки информационных систем, включая модели и методологии разработки, а также принципы тестирования программного обеспечения. Также представлена характеристика организации — Колледжа АлтГУ.

Вторая глава посвящена практической части работы: проектированию системы, её реализации с использованием технологического стека ASP.NET Core MVC, Entity Framework Core и PostgreSQL, тестированию и документированию.

Курсовая работа состоит из введения, двух глав, заключения, списка использованных источников. Работа содержит 3 рисунка, 2 таблицы.

## 1.2. Основы разработки информационных систем {#12-основы-разработки-информационных-систем}

### 1.2.1. Понятие и классификация информационных систем

Информационная система (ИС) представляет собой совокупность технических, программных и информационных средств, а также персонала, предназначенную для сбора, хранения, обработки, передачи и отображения информации с целью поддержки процессов принятия решений в различных сферах человеческой деятельности [1].

Классификация информационных систем может быть осуществлена по различным критериям. По масштабу ИС подразделяются на:

- **Персональные** — предназначены для работы одного пользователя (Microsoft Excel, текстовые редакторы)
- **Групповые** — обеспечивают работу группы пользователей в рамках одного подразделения (системы документооборота)
- **Корпоративные** — охватывают организацию в целом и интегрируют различные бизнес-процессы (ERP-системы)

По уровню управления выделяют:

- **Оперативные системы** — обрабатывают текущие операции и транзакции
- **Тактические системы** — поддерживают принятие решений на уровне управления
- **Стратегические системы** — обеспечивают долгосрочное планирование и стратегическое управление

По области применения ИС классифицируют на:

- Экономические и финансовые
- Управления производством
- Маркетинговые
- Кадровые
- Образовательные
- Медицинские и другие

### 1.2.2. Модели разработки информационных систем

Разработка информационных систем может осуществляться с использованием различных моделей, определяющих последовательность этапов и подход к управлению проектом.

**Каскадная модель** (waterfall) предполагает последовательное выполнение этапов разработки: анализ требований, проектирование, реализация, тестирование, внедрение и сопровождение. Каждый этап должен быть полностью завершён перед началом следующего. Достоинствами данной модели являются чёткая структура и понятные критерии перехода между этапами. Недостатками считаются сложность внесения изменений на поздних этапах и длительный цикл разработки [2].

**Инкрементная модель** предполагает разработку системы через последовательность версий, каждая из которых добавляет новую функциональность. Это позволяет раньше получить работающий продукт и учитывать изменения требований.

**Спиральная модель** объединяет преимущества каскадной и инкрементной моделей. Разработка осуществляется итерациями, каждая из которых включает анализ рисков, проектирование, реализацию и оценку результатов. Модель особенно эффективна для крупных и сложных проектов с высоким уровнем риска.

### 1.2.3. Гибкие методологии разработки

**Agile** — это семейство гибких методологий разработки, основанных на принципе итеративной разработки и сотрудничестве между саморганизующимися кросс-функциональными командами. Манифест Agile провозглашает следующие ценности [3]:

- Люди и взаимодействие важнее процессов и инструментов
- Работающий продукт важнее исчерпывающей документации
- Сотрудничество с заказчиком важнее контрактных условий
- Готовность к изменениям важнее следования первоначальному плану

**Scrum** — одна из наиболее популярных гибких методологий, использующая понятие спринтов — коротких итераций обычно продолжительностью 2-4 недели. В рамках Scrum выделяют следующие роли:

- Product Owner — владелец продукта, определяющий приоритеты
- Scrum Master — фасилитатор процесса
- Development Team — команда разработки

Scrum предполагает проведение ежедневных совещаний (daily standup), планирования спринта, обзора спринта и ретроспективы.

**DevOps** — методология, объединяющая разработку (Development) и эксплуатацию (Operations), направленная на сокращение жизненного цикла разработки и обеспечение непрерывной доставки ПО. Ключевые практики DevOps включают непрерывную интеграцию (CI), непрерывную доставку (CD) и инфраструктуру как код (IaC).

### 1.2.4. Жизненный цикл программного обеспечения

Жизненный цикл программного обеспечения (Software Development Life Cycle, SDLC) включает следующие основные этапы:

1. **Анализ требований** — определение функциональных и нефункциональных требований к системе, документирование в виде спецификации требований.

2. **Проектирование** — создание архитектуры системы, проектирование баз данных, определение структуры модулей и интерфейсов.

3. **Реализация** — написание программного кода согласно проектной документации.

4. **Тестирование** — проверка соответствия системы требованиям, выявление и исправление дефектов.

5. **Внедрение** — развёртывание системы в производственной среде, обучение пользователей.

6. **Сопровождение** — поддержка работоспособности системы, исправление ошибок, внесение изменений.

Каждый из этих этапов важен для создания качественного программного продукта и должен выполняться с должным уровнем внимания и тщательности.

### 1.2.5. Инструменты проектирования информационных систем

Для проектирования информационных систем используются различные инструменты, позволяющие визуализировать структуру системы и документировать проектные решения.

**UML (Unified Modeling Language)** — унифицированный язык моделирования, являющийся стандартом для визуализации, спецификации, конструирования и документирования программных систем. Основные диаграммы UML включают [4]:

- Диаграмма классов (Class Diagram) — отображает структуру системы в терминах классов и их отношений
- Диаграмма компонентов (Component Diagram) — показывает компоненты системы и их зависимости
- Диаграмма последовательности (Sequence Diagram) — демонстрирует взаимодействие объектов во времени
- Диаграмма прецедентов (Use Case Diagram) — описывает функциональные требования с точки зрения пользователей

**ER-модель (Entity-Relationship)** — модель данных, позволяющая описать концептуальную схему базы данных в терминах сущностей, их атрибутов и связей между ними. Основные компоненты ER-модели:

- **Сущность** — объект, информация о котором хранится в базе данных
- **Атрибут** — свойство сущности
- **Связь** — ассоциация между сущностями

Для проектирования баз данных используются такие инструменты, как:

- **Microsoft Access** — СУБД с визуальным проектированием
- **MySQL Workbench** — инструмент для проектирования и администрирования MySQL
- **pgAdmin** — инструмент администрирования PostgreSQL
- **DbVisualizer** — кроссплатформенный инструмент для работы с различными СУБД
- **DBeaver** — универсальный клиент для работы с базами данных

### 1.2.6. Технологический стек разработки

Современная веб-разработка предполагает использование различных технологий и инструментов. **ASP.NET Core** — кроссплатформенный фреймворк от Microsoft для создания веб-приложений, предоставляющий мощные средства для разработки, тестирования и развёртывания приложений [5].

**Entity Framework Core** — объектно-реляционный mapper (ORM), позволяющий работать с базой данных через объектно-ориентированный интерфейс, упрощая выполнение операций CRUD (Create, Read, Update, Delete).

**PostgreSQL** — объектно-реляционная система управления базами данных с открытым исходным кодом, известная своей надёжностью и расширяемостью.

**Bootstrap** — фреймворк для разработки адаптивных и мобильных веб-сайтов, предоставляющий набор HTML- и CSS-шаблонов, а также JavaScript-компонентов.

## 1.3. Принципы тестирования программ {#13-принципы-тестирования-программ}

### 1.3.1. Основные понятия тестирования программного обеспечения

Тестирование программного обеспечения представляет собой процесс исследования, анализа и оценки качества программного продукта с целью выявления дефектов и.verify соответствия требованиям. Качество программного обеспечения включает такие характеристики, как функциональность, надёжность, производительность, удобство использования и безопасность [6].

**Дефект** (bug) — это несоответствие между фактическим и ожидаемым поведением программы. Дефекты могут возникать на различных этапах разработки вследствие ошибок проектирования, кодирования или конфигурации.

**Тестовый случай** (test case) — это документ, описывающий входные данные, условия выполнения и ожидаемый результат для проверки определённой функциональности.

**Тестовое покрытие** — метрика, показывающая, какая часть кода или требований была проверена тестами. Различают покрытие кода (code coverage) и покрытие требований (requirements coverage).

### 1.3.2. Виды тестирования

Тестирование программного обеспечения классифицируется по различным критериям.

**По уровню тестирования:**

- **Модульное (юнит-тестирование)** — тестирование отдельных модулей или компонентов программы в изоляции от других частей системы. Модульное тестирование позволяет выявить ошибки на ранних этапах и упрощает отладку [7].

- **Интеграционное** — тестирование взаимодействия между отдельными модулями или компонентами системы. Целью является обнаружение ошибок в интерфейсах и взаимодействии частей системы.

- **Системное** — тестирование системы в целом на соответствие функциональным и нефункциональным требованиям.

- **Приёмочное** — проводится заказчиком для определения готовности системы к эксплуатации.

**По целям тестирования:**

- **Функциональное тестирование** — проверка соответствия системы функциональным требованиям.

- **Нефункциональное тестирование** — оценка нефункциональных характеристик: производительности, безопасности, удобства использования, масштабируемости.

- **Регрессионное тестирование** — повторное выполнение тестов после внесения изменений для обнаружения новых дефектов.

### 1.3.3. Тестирование методом чёрного и белого ящика

**Тестирование методом чёрного ящика** (Black Box Testing) — подход, при котором тестирование выполняется без знания внутренней структуры программы. Тесты основываются только на спецификации требований. Метод позволяет проверять функциональность с точки зрения пользователя, но не может гарантировать покрытие всех ветвей кода [8].

Преимущества тестирования чёрного ящика:

- Не требует знания внутренней структуры кода
- Тесты могут разрабатываться параллельно с разработкой кода
- Позволяет выявить несоответствия в требованиях

Недостатки:

- Ограниченное покрытие кода
- Сложность идентификации причины дефекта

**Тестирование методом белого ящика** (White Box Testing) — подход, при котором тестирование выполняется с полным знанием внутренней структуры программы. Тесты разрабатываются на основе анализа кода, структуры данных и алгоритмов.

Преимущества тестирования белого ящика:

- Возможность достижения высокого покрытия кода
- Раннее обнаружение дефектов
- Оптимизация кода

Недостатки:

- Требует знания кода и навыков программирования
- Не выявляет ошибки в требованиях

### 1.3.4. Тестирование веб-приложений

Веб-приложения имеют свою специфику, обусловленную архитектурой «клиент-сервер», использованием сетевых протоколов и разнообразием клиентских устройств и браузеров.

**Функциональное тестирование веб-приложений** включает проверку:

- Корректности работы всех функций и модулей
- Навигации и пользовательского интерфейса
- Обработки форм и ввода данных
- Аутентификации и авторизации пользователей
- Работы с базой данных

**Тестирование совместимости** (Cross-browser Testing) — проверка корректности работы приложения в различных браузерах (Chrome, Firefox, Safari, Edge) и на различных устройствах.

**Тестирование безопасности** — выявление уязвимостей, связанных с:

- SQL-инъекциями
- XSS (межсайтовым скриптингом)
- CSRF (подделкой межсайтовых запросов)
- Небезопасной аутентификацией и авторизацией

**Тестирование производительности** включает:

- Нагрузочное тестирование — проверка поведения системы под ожидаемой нагрузкой
- Стресс-тестирование — определение пределов работоспособности системы
- Тестирование отклика — измерение времени отклика системы

### 1.3.5. Тестирование методом серого ящика

**Тестирование методом серого ящика** (Grey Box Testing) — комбинированный подход, при котором тестировщик имеет частичное знание внутренней структуры системы. Данный метод сочетает преимущества чёрного и белого ящика, позволяя эффективно тестировать сложные системы.

### 1.3.6. Чек-лист тестирования веб-приложения

Для систематизации процесса тестирования используются чек-листы — списки проверок, которые необходимо выполнить. Чек-лист тестирования веб-приложения включает следующие категории [9]:

5. После успешной регистрации выполните вход

**Прохождение теста:**

1. Войдите в систему
2. Перейдите на страницу «Тесты»
3. Выберите тест из списка
4. Ответьте на все вопросы теста
5. Нажмите «Отправить»
6. Просмотрите результаты

**Создание теста (для администратора):**

1. Войдите в систему с правами администратора
2. Перейдите в панель администратора
3. Нажмите «Создать тест»
4. Заполните название и описание теста
5. Добавьте вопросы, укажите тип и варианты ответов
6. Нажмите «Сохранить»

# ЗАКЛЮЧЕНИЕ {#заключение}

В ходе выполнения курсовой работы было разработано веб-приложение «Testosteron» для проведения тестирования в образовательных учреждениях. В соответствии с поставленной целью были выполнены следующие задачи:

1. Изучены теоретические основы разработки информационных систем, включая модели и методологии разработки (каскадная, инкрементная, Agile, Scrum, DevOps), а также принципы тестирования программного обеспечения.

2. Проведён анализ требований к разрабатываемой системе и сформулированы функциональные требования.

3. Спроектирована архитектура системы на основе трёхуровневой модели, разработана ER-модель базы данных и создан проект с использованием UML-диаграмм.

4. Реализовано веб-приложение с использованием современного технологического стека: ASP.NET Core MVC 8.0, Entity Framework Core 8.0, PostgreSQL, Bootstrap 5.

5. Проведено тестирование разработанного приложения с использованием функционального тестирования и чек-листов.

6. Составлена документация и руководство пользователя.

**Технические характеристики разработанной системы:**

- Платформа: веб-приложение
- Архитектура: трёхуровневая (MVC)
- База данных: PostgreSQL с использованием Entity Framework Core
- Аутентификация: ASP.NET Core Identity
- Адаптивный интерфейс: Bootstrap 5

**Функциональные возможности:**

- Создание тестов с различными типами вопросов
- Прохождение тестов пользователями
- Аутентификация и авторизация
- Разделение прав доступа (администратор, пользователь)

**Дальнейшее развитие системы** может включать:

- Добавление статистики и аналитики
- Импорт/экспорт тестов
- Ограничение времени на прохождение теста
- Отправка результатов на email
- Мобильное приложение

Разработанное приложение соответствует современным требованиям к программному обеспечению и может быть использовано в образовательном процессе Колледжа АлтГУ для проведения тестирования студентов.
